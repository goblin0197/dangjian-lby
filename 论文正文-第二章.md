# 第二章 相关技术介绍

## 2.1 Spring Boot框架

本系统后端采用Spring Boot 2.7.2作为核心开发框架。Spring Boot基于Spring框架的自动配置机制，通过`@SpringBootApplication`注解即可启动一个内嵌Tomcat服务器的独立应用，无需额外部署外部Servlet容器。项目在pom.xml中引入`spring-boot-starter-web`依赖即可获得完整的Web开发能力，包括RESTful接口支持、JSON序列化和参数校验等功能。ORM层选用MyBatis-Plus 3.5.2替代原生MyBatis，该框架在保留MyBatis灵活SQL编写能力的同时提供了BaseMapper通用CRUD方法、分页插件、逻辑删除和自动填充等开箱即用功能，显著减少了样板代码的编写量。系统控制器层统一使用`@RestController`注解标注（如`UserController`、`OrganizationController`、`ActivityController`），配合`@RequestMapping`定义URL路径前缀。服务层通过`@Service`注解注册Bean（如`UserService`、`OrganizationService`、`DevelopmentStageService`），各Service之间按模块边界进行调用。

AOP切面机制在本系统中承担了两项关键职责。权限校验切面`AuthInterceptor`拦截带有`@AuthCheck(mustRole = {...})`注解的控制器方法，从当前会话中提取用户角色信息并与注解声明的允许角色列表做匹配校验，不满足条件时直接抛出权限异常。操作日志切面`DevelopmentStageLogAspect`则专门处理党员发展阶段的变更追踪——当控制器方法标注了`@DevelopmentStageLogAnnotation(operationType = CREATE/UPDATE/SUBMIT_AUDIT/AUDIT/DELETE)`注解时，切面会在方法执行前后分别捕获实体对象的快照数据，将变更前后的JSON差异记录写入`development_stage_log`表，实现了对关键业务操作的完整审计追溯能力。

## 2.2 MySQL数据库

MySQL 8.x作为本系统的关系型数据存储引擎，承载全部核心业务数据的持久化工作。数据库名为`party_building`，共包含16张数据表，覆盖用户管理、组织架构、发展阶段、活动报名、材料提交、公告发布、量化统计等全部业务域。其中`user`表存储15个字段的用户基础信息和角色属性，`organization`表通过parentId字段实现树形层级结构的自关联存储，`development_stage`表以16个字段记录党员发展各阶段的时间节点、考察内容和审核状态，`activity`表包含20个字段支撑活动的完整生命周期管理，`material_submission`表以22个字段维护材料的多状态流转过程，`user_quantify`和`org_quantify`两张表分别存储用户维度和组织维度的量化统计数据。

MyBatis-Plus框架为数据库访问提供了两项重要的基础设施配置。全局ID策略设置为自增模式（AUTO），所有主键字段由MySQL数据库自动生成递增值。逻辑删除机制启用`isDelete`字段（0表示未删除、1表示已删除），框架在执行查询和更新操作时自动追加该字段的过滤条件，确保已标记删除的数据不会出现在业务查询结果中，同时保留了物理数据用于后续审计或恢复操作。

## 2.3 Redis缓存中间件

Redis在本系统中主要承担分布式会话管理的职责。系统通过Spring Session集成Redis作为会话存储后端，将用户的登录状态和Session属性序列化后写入Redis服务器而非本地内存。配置文件中指定Redis连接地址为localhost:6379，使用database索引2隔离党建系统的会话数据与其他应用的缓存空间。会话过期时间设定为30天，在此期间用户无需重复登录即可保持在线状态。当系统未来扩展至多实例部署场景时，基于Redis的共享Session机制能够保证用户请求路由到任意实例都能正确识别其登录身份，无需引入额外的Session同步方案。

除会话管理外，Redis还为量化统计模块预留了热点数据缓存能力。量化统计涉及多表聚合计算（活动报名数、签到数、材料文件数的COUNT与SUM运算），在高并发查询场景下可能对MySQL造成较大压力。将统计结果缓存至Redis并设置合理的过期时间（如每小时刷新一次），可大幅降低数据库负载。当前版本中此优化尚未完全实现，但`QuantifyScheduledTask`定时任务的架构设计已为此类缓存策略的接入预留了扩展点。

## 2.4 Vue.js框架

前端界面采用Vue 3.2.13构建，选择该版本的核心原因在于其对组合式API（Composition API）的原生支持。组合式API允许开发者将组件中的响应式状态、计算属性和方法按照逻辑关注点组织到`setup()`函数内部，相比Vue 2的Options API能够更灵活地实现跨组件的逻辑复用。项目全局使用TypeScript 4.5.5进行类型约束，所有API接口返回值、Vuex Store状态定义和Props类型均声明了严格的类型签名，编译阶段即可发现大部分类型错误。页面组件按功能模块组织在`src/views/`目录下：`views/user/UserLoginView.vue`负责登录交互、`views/organization/OrganizationView.vue`呈现党组织树形结构、`views/activity/ActivityView.vue`管理活动全生命周期、`views/trainerRelation/TrainerRelationView.vue`展示发展流程与审核操作、`views/quantify/QuantifyManagerView.vue`和`views/quantify/QuantifyView.vue`分别提供管理员和个人视角的统计图表。

各模块的HTTP请求封装在`src/api/`目录下的独立TypeScript文件中（如`api/yonghuguanli.ts`、`api/dangzuzhiguanli.ts`、`api/huodongguanli.ts`），每个文件导出对应模块的全部API调用函数，函数内部通过Axios实例发起请求并统一处理响应码判断和错误弹窗提示。全局状态由Vuex 4.0管理，`store/user.ts`维护登录状态标识和当前用户对象，`store/headerMenu.ts`控制顶部导航菜单的展开折叠行为。前端路由守卫结合`access/checkAccess.ts`中的权限枚举`ACCESS_ENUM`实现页面级访问控制，未授权角色访问受限路由时自动跳转至无权限提示页。

## 2.5 ECharts可视化库

ECharts 6.0.0是量化统计模块的数据可视化渲染引擎。该库由百度开源团队维护，提供柱状图、折线图、饼图、雷达图、散点图等三十余种标准图表类型，支持数据缩放、工具提示、图例筛选、动画过渡等丰富的交互特性。本系统中ECharts被应用于两个页面的图表展示：管理员视角的`QuantifyManagerView.vue`页面展示组织维度的统计报表，包括各支部活动次数对比的柱状图、月度报名人数趋势的折线图以及活动类型分布的饼图；普通用户视角的`QuantifyView.vue`页面呈现个人维度的三维评估雷达图，坐标轴分别映射参与率、签到率和材料完成率三个指标。

图表的数据绑定流程为：前端调用`api/lianghuatongji.ts`中封装的量化统计查询接口获取后端聚合数据，在组件的`onMounted`生命周期钩子中将数据传入ECharts实例的`setOption()`方法完成图表初始化。管理员页面还支持按时间范围和组织层级切换统计粒度，切换时重新触发接口请求并更新图表配置项。ECharts的响应式布局能力确保了图表在不同分辨率屏幕下的自适应显示效果。

## 2.6 本章小结

本章介绍了智慧党建综合管理系统所采用的全部关键技术。后端Spring Boot生态提供了稳定的RESTful API服务能力和AOP切面式的横切关注点处理机制；MySQL数据库以16张表的结构化设计支撑全部业务域的数据持久化需求；Redis保障了多实例部署场景下会话状态的一致性；前端Vue 3 + TypeScript组合实现了类型安全的组件化开发和灵活的状态管理；ECharts为量化统计模块提供了专业的数据可视化渲染能力。整套技术栈各组件间协同性好、社区文档丰富、学习曲线平缓，能够充分满足本系统的功能开发和性能要求。
