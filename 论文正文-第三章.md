# 第三章 系统需求分析

## 3.1 可行性分析

### 3.1.1 技术可行性

本系统所选技术栈在工业界和学术界均经过了大规模生产环境的验证。Spring Boot框架自2014年发布以来已成为Java企业级开发的事实标准，其自动配置机制和起步依赖体系将传统Spring项目的XML配置量压缩了百分之八十以上，开发者可专注于业务逻辑编写而非环境搭建。MyBatis-Plus作为MyBatis的增强工具在国内开源社区拥有广泛的用户基础，其分页插件、代码生成器和通用CRUD接口已被大量项目采用。Vue 3于2020年正式发布后，组合式API的设计理念得到了前端社区的普遍认可，TypeScript的类型安全能力也在中大型项目中展现出降低运行时错误率的实际效果。MySQL 8.x在单表千万级数据量下的查询性能表现稳定，配合合理的索引设计完全能够支撑高校党组织规模（通常为数百至数千名党员）的数据存储需求。团队成员具备Java Web全栈开发和Vue组件化编程的基础能力，参考同类智慧党建系统[3][7]的技术路线，系统在技术层面不存在难以逾越的实现障碍。

### 3.1.2 经济可行性

从开发成本角度评估，系统所依赖的全部核心技术组件均为开源软件。Spring Boot、Vue.js、MySQL Community Edition、Redis、ECharts等均可免费下载使用，无需支付任何商业授权费用或订阅费用。文件存储环节选用腾讯云COS对象存储服务，按实际存储量和请求次数计费，对于党建系统的文档类小文件存储场景而言月度成本极低。硬件部署方面，一台配置4核CPU、8GB内存、100GB SSD的标准云服务器即可同时运行Spring Boot后端服务和Nginx静态资源代理，年租金约数百元人民币。部署运维环节得益于Spring Boot的内嵌Tomcat机制，整个应用打包为一个独立JAR文件，通过`java -jar`命令即可启动运行；前端构建产物为纯静态HTML/CSS/JavaScript文件，可直接托管在Nginx或CDN节点上。Docker容器化部署可将应用和环境依赖打包为镜像，实现一键启动和横向扩展。对比传统纸质化管理模式下人工统计活动数据、手工归档材料文档、线下审批组织关系转移所耗费的人力工时，信息化系统能够将上述流程的处理效率提升数倍以上，长期运营成本显著低于人力投入。

## 3.2 系统功能需求分析

### 3.2.1 用户管理模块需求

系统应提供基于账号密码的用户登录与注销功能，登录时对密码进行MD5加密校验，验证通过后将用户身份信息写入Session并返回包含角色编码的登录凭证。管理员角色（super_admin和org_admin）应具备用户的创建、编辑、删除和查询权限，创建用户时系统自动分配默认密码并进行加密存储。系统需支持四种角色的权限隔离：超级管理员拥有全部功能的操作权限；党支部管理员仅能管理所属支部范围内的组织和成员数据；普通党员可参与活动报名、提交材料、查看个人统计数据；入党积极分子和发展对象除上述功能外还需支持发展材料的上传操作。普通用户应能够修改个人基本信息（姓名、手机号、邮箱）和登录密码。系统应在所有涉及密码传输和存储的环节执行MD5哈希处理。

### 3.2.2 党组织管理模块需求

系统应能够维护党委—党总支—党支部三级树形层级结构的党组织架构，每个组织节点记录组织名称、唯一编码、父组织指针、组织类型和组织级别等信息。管理员应能执行组织的创建、编辑、删除和查询操作，删除组织前系统需检查是否存在下属子组织或已关联成员以防止数据孤立。系统应支持为指定组织绑定负责人，绑定后在组织列表中展示负责人姓名。组织成员管理功能允许管理员查看某组织下全部成员列表并执行成员调入调出操作。党员组织关系转移功能应覆盖申请发起、审批处理、状态变更的完整流程：党员在线提交转移申请并填写原组织、目标组织和转移原因，管理员审核后执行通过或拒绝操作，审批意见和审批时间需完整记录。组织查询结果应以树形结构呈现以便直观展示层级关系。

### 3.2.3 发展阶段模块需求

系统应支持对党员发展四个阶段（入党积极分子、发展对象、预备党员、正式党员）的全流程跟踪管理。每个阶段记录应包含阶段名称、起止时间、考察内容、培养人信息和当前状态等属性。培养人关联功能允许为被培养对象指定一名或多名正式党员作为培养联系人，记录培养关系的开始日期、结束日期和关系状态。阶段提交审核功能使积极分子或管理员能够将处于进行中的阶段记录提交至审核流程。审核操作由具有相应权限的管理员执行，审核时填写考察结果（合格或不合格）及审核意见，审核通过的记录标记为已完成并可进入下一阶段。系统应通过AOP切面机制自动捕获阶段记录的创建、更新、提交审核、审核和删除五类操作，将操作类型、操作人、变更前后数据快照写入日志表供后续追溯查询。阶段状态应支持进行中、已完成和已终止三种状态的流转控制。

### 3.2.4 活动管理模块需求

系统应支持四种类型的活动创建：会议、志愿活动、学习培训和其他自定义类型。活动记录应包含活动名称、所属组织、活动内容描述、时间地点安排、最大参与人数限制等基本信息。活动的生命周期应经历待发布、已发布、进行中和已结束四个状态阶段，管理员可在各阶段间执行状态转换操作。活动报名功能允许普通用户在报名截止时间内提交报名请求，系统实时校验人数上限并在达到限制后拒绝新增报名。签到管理应支持两种方式：扫码签到（生成活动专属二维码供参与者扫描确认）和手动签到（管理员在后台逐条标记签到状态）。系统应能够根据报名记录中的签到标记自动计算每场活动的总报名人数、实际参与人数和签到率百分比。活动结束后管理员可填写活动回顾总结内容存档。

### 3.2.5 材料管理模块需求

超级管理员应能够按发展阶段分类上传和管理材料模板文件，模板文件存储至腾讯云COS对象存储服务，模板支持启用和停用状态切换。入党积极分子和发展对象应根据已启用的模板要求上传个人发展材料文件，提交时选择对应的模板和发展阶段。材料审核分为两个层级：初审由支部管理员（org_admin角色）负责执行，终审由超级管理员（super_admin角色）负责执行。初审不通过的材料退回给提交者并附注修改意见，提交者可修正后重新提交。初审通过的材料进入终审队列等待超级管理员复核。终审通过后的材料系统自动执行归档操作，将材料元数据和文件路径写入归档表并记录归档人和归档时间。已归档材料应支持按用户、阶段、类型等条件的检索和查看。系统首页应展示当前用户的待提交材料数量和待审核材料数量提醒。

### 3.2.6 公告管理模块需求

系统应区分系统公告和支部公告两种范围：系统公告面向全体注册用户可见，支部公告仅限该组织及其下属组织的成员可见。公告创建时应支持富文本编辑器编写正文内容，设置标题、摘要、封面图和附件等属性。公告发布功能将草稿状态的公告切换为已发布状态并记录发布时间。已发布公告应支持撤回操作（恢复为草稿状态）、下架操作（标记为过期不可见）和重新发布操作。置顶管理功能允许管理员将重要公告固定在列表顶部显示。批量删除功能支持一次选中多条公告执行软删除操作。公告详情页展示完整的富文本正文内容和附件下载链接。已读追踪机制在用户点击查看公告详情时写入阅读记录，系统据此计算每位用户的未读公告数量用于首页红点提示。

### 3.2.7 量化统计模块需求

系统应能够从用户维度和个人维度分别生成量化统计数据。用户维度统计指标包括活动参与率（已报名活动数占总活动数的比例）、签到率（已签到数占已报名数的比例）和材料完成率（已归档材料数占应提交材料总数的比例）。组织维度统计指标包括活动举办总数、累计报名人数、累计签到人数以及各项指标的支部平均值。统计数据应支持定时自动生成和手动触发两种触发方式：定时任务按照配置的Cron表达式周期性执行全量重算，管理员也可在界面上手动点击按钮即时刷新。统计结果通过ECharts图表库在前端页面渲染展示，管理员视角页面呈现组织维度的柱状图和折线图报表，普通用户视角页面展示个人维度的雷达图或多维饼图。量化指标的定义应支持可配置管理，管理员可增删统计指标项并设定其适用维度和适用组织层级范围。

## 3.4 本章小结

本章从技术可行性和经济可行性两个方面论证了系统开发的合理性。技术层面所采用的Spring Boot + Vue.js + MySQL技术栈均为成熟稳定的开源方案，团队具备相应的开发能力且同类项目的成功案例提供了参考依据。经济层面全部核心组件零授权成本、硬件要求低廉、部署运维简便，相比传统管理模式的人力投入具备显著的性价比优势。功能需求部分围绕七大业务模块逐一明确了系统应具备的能力边界，涵盖用户认证与权限控制、组织架构与关系转移、发展流程跟踪与日志审计、活动生命周期与签到统计、材料多级审核与自动归档、公告发布与已读追踪、量化统计与可视化展示等全部核心业务场景，为后续的系统设计和代码实现工作提供了清晰的功能规格输入。
