# 第四章 系统设计

## 4.1 系统总体架构设计

本系统采用经典的前后端分离B/S架构模式，整体划分为四个逻辑层次。浏览器表示层运行在用户终端的Chrome或Edge等现代浏览器中，由Vue 3单页应用（SPA）承担页面渲染和用户交互职责。该层使用Ant Design Vue 4.2.6组件库构建表单、表格、弹窗等UI元素，ECharts 6.0负责统计图表的画布渲染，Vuex 4.0管理全局状态，Vue Router 4.0控制页面路由跳转。前端通过Axios HTTP客户端向后端发起RESTful风格的数据请求，请求统一携带Session Cookie以标识当前登录用户身份。

API网关层位于前后端之间承担请求转发和负载均衡职能。本系统的Spring Boot应用内嵌Tomcat服务器监听8101端口，上下文路径配置为`/api`。所有外部请求经Nginx反向代理（可选部署环节）转发至后端服务，Nginx同时负责静态资源的托管和Gzip压缩传输优化。后端业务层是系统的核心处理单元，基于Spring Boot 2.7.2框架构建，遵循MVC分层模式组织代码：Controller层接收HTTP请求并完成参数校验与路由分发，Service层封装具体业务逻辑和事务边界，Mapper层通过MyBatis-Plus执行SQL操作并与MySQL数据库交互。AOP切面横切于Controller层与Service层之间，分别实现权限拦截校验和操作日志自动记录两项横切关注点。

数据存储层包含三个独立存储节点。MySQL 8.x数据库实例（数据库名party_building）承载全部结构化业务数据的持久化存储，共16张表按业务域划分。Redis服务器（localhost:6379，database索引2）存储Spring Session序列化后的会话数据，保障多实例部署时登录状态的共享一致性。腾讯云COS对象存储服务接收前端上传的材料文件、模板文件和用户头像等二进制资源，返回的URL地址存入MySQL对应记录中供后续访问引用。

## 4.2 系统功能模块设计

系统按照业务领域划分为七大功能模块，各模块在职责边界上保持高内聚、在交互接口上保持低耦合。

用户管理模块作为全系统的基础设施模块，提供身份认证服务和账户生命周期管理能力。其他六个模块均依赖该模块输出的用户身份信息（当前登录用户ID、角色编码、所属组织ID）来执行权限判断和数据过滤。该模块的输出为经过认证的User会话对象和角色枚举值。

党组织管理模块维护系统内的组织层级树形结构。该模块为活动管理模块提供活动所属组织的上下文信息（orgId），为材料管理模块限定材料提交的组织范围，为量化统计模块提供组织维度的聚合计算粒度。组织关系转移子模块独立处理跨组织的党员调动审批流程，其输入为用户的转移申请、输出为组织归属关系的变更结果。

发展阶段模块是本系统的核心特色模块，实现了党员发展全流程的数字化跟踪能力。该模块与用户管理模块关联（每个阶段记录绑定一个userId）、与党组织管理模块间接关联（通过用户所属组织确定阶段审核的管理范围）。材料管理模块依赖该模块的阶段分类属性来组织模板和提交材料的归类展示。该模块内部集成的AOP日志切面机制是其区别于同类系统的关键设计亮点。

活动管理模块支撑党建活动的完整运营流程。该模块产生的报名数据和签到数据是量化统计模块的核心数据来源——参与率、签到率等指标均从活动及其关联的报名记录表中聚合计算得出。活动与党组织的多对一关系确保了每场活动都有明确的组织归属。

材料管理模块实现发展材料的模板化管理和多级审核归档流程。该模块依赖发展阶段模块的阶段定义来分类组织模板库，依赖用户管理模块的身份信息来判断当前用户的操作权限（超级管理员管理模板、积极分子提交材料、支部管理员初审、超级管理员终审）。腾讯云COS集成是该模块的外部依赖项。

公告管理模块相对独立，主要承担信息的发布与推送职责。公告的组织可见性范围依赖于党组织模块的组织成员列表来确定目标受众群体。已读追踪功能需要结合用户身份来标记阅读记录和统计未读数量。

量化统计模块作为数据消费型模块，从活动管理、材料管理和发展阶段等多个上游模块汇聚原始数据，经过聚合运算后生成多维度的统计指标。该模块不产生新的业务数据变更，仅提供只读查询和分析展示能力。ECharts图表渲染是该模块的前端技术依赖。

## 4.3 数据库设计

### 4.3.1 数据库概念结构设计

数据库概念模型识别出以下核心实体及其相互关系。User实体代表系统注册用户，其主键id被多个外键引用：Organization实体通过leaderId字段指向User表示负责人绑定关系；DevelopmentStage实体通过userId字段指向User表示阶段记录归属；ActivityEnroll实体通过userId字段指向User表示报名记录归属；MaterialSubmission实体通过user_id字段指向User表示材料提交者；TrainerRelation实体通过userId和trainerId两个字段分别指向User表示培养关系的双方；UserQuantify实体通过userId字段指向User表示统计数据归属。Organization实体存在自关联关系——parentId字段指向另一条Organization记录的id，形成党委到党总支再到党支部的树形层级链条。DevelopmentStage实体通过trainerId字段关联TrainerRelation中的培养人信息。Activity实体通过orgId字段关联所属组织，ActivityEnroll实体通过activityId字段关联对应的活动记录。MaterialSubmission实体同时关联User（提交者）和DevelopmentStage（所属阶段）两个父实体。Notice实体的orgId字段可选关联Organization（为null时表示系统级全员公告）。OrgQuantify实体通过orgId字段关联组织维度统计数据。

上述实体关系中，一对多关系占主导地位：一个组织拥有多名用户、一名用户可有多条发展阶段记录、一场活动可有多条报名记录、一个用户可提交多份材料。多对多关系通过中间表解耦（如培养人关系通过独立的trainer_relation表维护）。全部实体均采用bigint类型自增主键确保唯一性标识。

### 4.3.2 数据库物理结构设计

**（1）user表**

user表存储系统中全部注册用户的基础信息和身份数据，共计15个字段。该表的设计要点如下：

| 字段名 | 类型 | 是否主键 | 说明 |
|--------|------|----------|------|
| id | bigint | 是 | 主键，自增 |
| userAccount | varchar(256) | 否 | 登录账号，设置唯一约束 |
| userPassword | varchar(512) | 否 | MD5加密密码 |
| userName | varchar(256) | 否 | 用户姓名 |
| userAvatar | varchar(1024) | 否 | 头像URL（COS路径） |
| userRole | varchar(256) | 否 | 角色编码 |
| orgId | bigint | 否 | 所属党组织ID，外键 |
| phone | varchar(32) | 否 | 手机号 |
| email | varchar(256) | 否 | 邮箱 |
| userType | varchar(64) | 否 | 用户类型（教师/学生） |
| status | tinyint | 否 | 账号状态（0正常/1禁用） |
| joinDate | date | 否 | 申请入党日期 |
| positiveDate | date | 否 | 转正日期 |
| politicalStatus | varchar(64) | 否 | 政治面貌 |

userRole字段的取值范围由UserRoleEnum枚举类严格限定为四种编码：super_admin（超级管理员）、org_admin（组织管理员）、party_member（党员）、activist_development（入党积极分子/发展对象）。控制器层的`@AuthCheck`注解依据此字段值进行接口级别的权限拦截。userPassword字段在写入数据库前必须经过`EncryptUtils.encryptPassword()`方法执行MD5哈希运算，明文密码不在任何环节持久化存储。orgId字段建立到organization表id列的外键约束，确保用户所属组织的引用完整性。joinDate和positiveDate两个字段用于记录党员发展的关键时间节点，配合development_stage表的时间区间字段形成完整的成长时间线。

**（2）organization表**

organization表维护党组织的树形层级结构，共计11个字段：

| 字段名 | 类型 | 是否主键 | 说明 |
|--------|------|----------|------|
| id | bigint | 是 | 主键，自增 |
| orgName | varchar(256) | 否 | 组织名称 |
| orgCode | varchar(64) | 否 | 组织编码，唯一约束 |
| parentId | bigint | 否 | 父组织ID（0表示顶级） |
| orgType | varchar(64) | 否 | 组织类型 |
| orgLevel | int | 否 | 组织级别 |
| leaderId | bigint | 否 | 负责人ID，外键→user.id |
| address | varchar(512) | 否 | 组织地址 |
| description | varchar(1024) | 否 | 组织描述 |

parentId字段是实现树形结构的核心设计——顶级组织（如校党委）的parentId设为0，下属党总支的parentId指向校党委的id值，党支部的parentId再指向对应党总支的id值。`OrganizationService.getSubGradedOrgs()`方法通过递归查询父子关系组装成`OrganizationGradedVO`嵌套对象返回给前端进行树形渲染。orgType字段区分党委、党总支和党支部三种组织类型，orgLevel字段记录数值化的层级深度供排序和筛选使用。leaderId字段建立到user表的外键关联，`fillOrganizationLeader()`查询方法在返回组织详情时自动JOIN填充负责人姓名。orgCode字段设置了数据库唯一约束防止重复创建。

**（3）development_stage表**

development_stage表是本系统最具特色的业务表，记录党员发展各阶段的详细跟踪数据，共计16个字段：

| 字段名 | 类型 | 是否主键 | 说明 |
|--------|------|----------|------|
| id | bigint | 是 | 主键，自增 |
| userId | bigint | 否 | 用户ID，外键 |
| trainerId | bigint | 否 | 培养人ID，外键 |
| stageName | varchar(256) | 否 | 阶段名称 |
| stageStartTime | date | 否 | 阶段开始时间 |
| stageEndTime | date | 否 | 阶段结束时间 |
| stageStatus | tinyint | 否 | 阶段状态 |
| assessmentContent | text | 否 | 考察内容 |
| assessmentResult | tinyint | 否 | 考察结果 |
| auditUserId | bigint | 否 | 审核人员ID |
| auditTime | datetime | 否 | 审核时间 |
| auditRemark | text | 否 | 审核意见 |

stageName字段限定四种取值：入党积极分子、发展对象、预备党员、正式党员，对应党员发展的标准程序各阶段。stageStatus字段构成简单的状态机：0表示进行中（默认初始状态）、1表示已完成（审核通过后设置）、2表示已终止（审核不合格或主动放弃时设置）。assessmentResult字段的三态取值含义为：1合格、0不合格、2未审核（默认值）。auditUserId和auditTime字段在执行审核操作时由系统自动填入当前操作人信息和服务器时间。该表的全部写操作（增删改及审核）均标注了`@DevelopmentStageLogAnnotation`注解触发AOP切面自动记录变更日志至`development_stage_log`表。

| 字段名 | 类型 | 是否主键 | 说明 |
|--------|------|----------|------|

| activityName | varchar(512) | 否 | 活动名称 |
| orgId | bigint | 否 | 所属组织ID |
| userId | bigint | 否 | 创建人ID |
| activityType | int | 否 | 活动类型编码 |
| activityContent | text | 否 | 活动描述 |
| enrollDeadline | datetime | 否 | 报名截止时间 |
| startTime | datetime | 否 | 活动开始时间 |
| endTime | datetime | 否 | 活动结束时间 |
| location | varchar(512) | 否 | 活动地点 |
| maxNum | int | 否 | 最大参与人数 |
| currentNum | int | 否 | 当前参与人数 |
| totalParticipant | int | 否 | 总报名人数 |
| actualParticipant | int | 否 | 实际参与人数 |
| signRate | double | 否 | 签到率百分比 |
| status | int | 否 | 活动状态 |
| signInType | int | 否 | 签到方式 |
| QRCodeUrl | varchar(1024) | 否 | 二维码URL |

activityType字段遵循`ActivityConstant`中定义的四类编码常量：1会议、2志愿活动、3学习、4其他。status字段定义活动的四态生命周期：1待发布（创建后默认状态）、2已发布（执行publish操作后）、3进行中（当前时间介于startTime和endTime之间）、4已结束（执行end操作或超时自动结束）。signInType字段区分两种签到模式：1扫码签到（生成QRCodeUrl供手机扫描）、2手动签到（管理员在后台逐条确认）。signRate字段由`updateActivityStatisticsById()`方法根据actualParticipant除以totalParticipant的计算结果自动更新，无需人工维护。totalParticipant、actualParticipant和signRate三个字段共同构成了量化统计模块的活动维度数据源。

| 字段名 | 类型 | 是否主键 | 说明 |
|--------|------|----------|------|
| id | bigint | 是 | 主键，自增 |
| userId | bigint | 否 | 用户ID，外键+唯一 |
| participateRate | double | 否 | 活动参与率 |
| signRate | double | 否 | 活动签到率 |
| fileCount | int | 否 | 已归档材料数 |
| activityCount | int | 否 | 参与活动总数 |
| signCount | int | 否 | 签到次数 |
| totalEnrollCount | int | 否 | 总报名次数 |
| statDate | date | 否 | 统计日期 |

userId字段建立了到user表的外键约束并同时设置了唯一约束，确保每位用户在同一统计日期下仅保留一条统计记录。participateRate、signRate和fileCount是三个核心展示指标，直接映射到前端ECharts雷达图的坐标轴维度。activityCount、signCount和totalEnrollCount是中间计算值，由`QuantifyDataService.generateUserQuantifyData()`方法从activity_enroll表和material_submission表中聚合COUNT运算得出。statDate字段支持按日期维度查看历史统计趋势，定时任务每次执行时以当前系统日期作为statDate值插入新记录或更新已有记录。

## 4.4 接口设计

系统后端对外暴露RESTful风格的HTTP API接口，全部接口统一挂载在`/api`上下文路径下。接口响应格式采用泛型包装类`BaseResponse<T>`统一封装，其中code字段表示业务状态码（0成功、非0失败）、data字段携带具体的返回数据、message字段附注错误提示信息。下面选取五个具有代表性的关键接口说明其契约规范。

用户登录接口POST `/api/user/login`是系统的入口接口。请求体为JSON格式的`UserLoginRequest`对象，包含userAccount（登录账号）和userPassword（登录密码）两个必填字段。后端`UserController.userLogin()`方法接收请求后调用`UserService.userLogin()`执行账号查找和密码比对（MD5加密后匹配），验证通过后将用户信息写入Redis Session并返回`LoginUserVO`对象，VO中包含用户ID、姓名、角色编码和组织ID等前端所需的基本信息。该接口无需任何角色权限即可匿名访问。

获取组织树形结构接口GET `/api/organization/get/graded?orgId=xxx`支持递归查询指定组织及其全部下属子组织的层级数据。请求参数orgId为可选参数，不传时从根组织开始遍历。`OrganizationService.getSubGradedOrgs()`方法执行递归SQL查询组装父子关系，返回`OrganizationGradedVO`嵌套树形对象，前端Ant Design Vue的Tree组件可直接消费该数据结构渲染可展开折叠的组织树。该接口要求调用者具备org_admin及以上角色权限。

添加发展阶段记录接口POST `/api/developmentStage/add`用于创建新的党员发展阶段跟踪记录。请求体为`DevelopmentStageAddRequest`对象，需传入userId、stageName、stageStartTime、stageEndTime、assessmentContent、trainerId等字段。该方法标注了`@DevelopmentStageLogAnnotation(operationType = CREATE)`注解，AOP切面会在方法执行完成后自动将新建记录的完整数据快照以JSON格式写入`development_stage_log`表。接口要求调用者具有管理员角色权限。

材料初审接口PUT `/api/material/submission/audit`实现材料的第一级审核操作。请求体包含材料记录ID、审核结果（通过/拒绝）和审核意见三个字段。后端校验当前登录用户角色是否为org_admin（支部管理员），校验通过后更新material_submission表的auditor、audit_time、audit_opinion和audit_status字段。若审核结果为拒绝则同步将submit_status回退至submitted状态允许用户重新提交。该接口是材料多级审核流程中的关键状态转换节点。

获取用户量化统计接口GET `/api/quantify/user/{userId}`查询指定用户的最新量化统计数据。路径参数userId指定目标用户ID。后端从user_quantify表中按userId条件查询最新一条记录，返回包含participateRate、signRate、fileCount等指标值的统计对象。若该用户尚无统计数据则触发一次即时计算后返回。该接口要求调用者具备管理员角色或为目标用户本人（数据权限校验）。
