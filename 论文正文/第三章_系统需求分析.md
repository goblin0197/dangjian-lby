# 第三章 系统需求分析

## 3.1 可行性分析

### 3.1.1 技术可行性

本系统所采用的后端技术栈以Spring Boot 2.7.2为核心框架，该框架自2014年发布以来经历了多个大版本的迭代优化，在国内外企业级应用开发中积累了大量成功案例和成熟的解决方案文档。MyBatis-Plus作为MyBatis的增强工具包提供了开箱即用的CRUD操作封装和分页插件，能够显著减少数据访问层的重复编码工作量。前端Vue 3框架配合TypeScript类型系统可以在编译阶段提前发现大部分代码逻辑错误，ECharts图表库的官方文档覆盖了近百种图表配置场景。上述技术组件均有活跃的开源社区持续维护，遇到技术问题时可通过GitHub Issues、Stack Overflow等渠道获取解决方案。开发团队对Java后端开发和Vue前端开发具备一定的实践经验储备，能够在项目周期内完成各功能模块的编码实现。

### 3.1.2 经济可行性

系统开发和运行所需的全部软件均为开源免费产品。Spring Boot、MyBatis-Plus、MySQL Community Server、Redis、Vue.js、Arco Design Vue、ECharts等核心组件均遵循Apache 2.0或MIT等宽松开源协议，不涉及商业许可费用。硬件层面，一台配置4核CPU、8GB内存、100GB硬盘的云服务器即可满足部署运行要求，按当前公有云市场的标准实例价格计算，年度服务器租赁成本在千元级别以内。开发工具方面IntelliJ IDEA提供社区免费版本，Visual Studio Code和Git均为免费软件。整体来看系统的开发和运维投入处于高校科研项目可承受的经济范围内。

### 3.1.3 操作可行性

系统采用B/S（浏览器/服务器）架构模式，终端用户仅需通过Chrome、Edge等主流浏览器访问系统网址即可使用全部功能，无需在本地计算机安装任何专用客户端程序或运行时环境。界面设计遵循常见的Web应用交互规范，左侧导航菜单、顶部搜索栏、表格列表页、表单弹窗页等布局形式与用户日常使用的各类管理后台系统保持一致的操作习惯，降低了用户的学习成本。管理员端的组织管理、活动发布、材料审核等操作均通过图形化界面完成，不涉及命令行输入或数据库直接操作，非技术人员经过简短培训即可独立完成日常事务处理。

## 3.2 系统功能需求分析

### 3.2.1 用户管理模块需求

系统应能够支持用户通过账号密码方式完成登录认证，登录成功后由服务端创建Session会话并写入Redis存储，后续请求携带会话凭证即可维持登录状态直至主动注销或超过30天过期时限。超级管理员应拥有创建新用户的权限，新建用户时系统自动设置默认密码为123456并通过EncryptUtils工具类进行MD5加密处理后存入user表的userPassword字段。系统应提供用户信息的查询、修改和删除功能，分页列表应支持按角色类型、所属组织、账号关键字等多维度条件组合筛选。系统应定义五类用户角色——超级管理员（super_admin）、组织管理员（org_admin）、党员（org_member）、发展对象（activist_development）和积极分子（activist），不同角色在接口访问权限上存在差异化的控制规则，权限校验通过@AuthCheck注解声明所需角色列表并由AuthInterceptor切面统一拦截验证。

### 3.2.2 党组织管理模块需求

系统应能够支持党组织的增删改查基本操作，每条组织记录包含组织名称、唯一编码、父组织ID、组织类型（党委/党总支/党支部）、组织级别、负责人ID、地址和描述信息等属性字段。系统应能够以树形层级结构展示从党委到党总支再到党支部的三级组织架构体系，当用户选中某一组织节点时应能递归加载其下属的全部子级组织形成完整的组织树视图。系统应提供组织负责人的绑定功能，管理员可从已审核通过的党员名单中选择一人作为某组织的负责人并将该党员的用户ID写入organization表的leaderId字段。组织的分页查询功能应支持按组织名称模糊匹配和组织类型精确筛选两种查询条件。

### 3.2.3 活动管理模块需求

系统应能够支撑党建活动的完整生命周期流转过程。活动创建阶段管理员填写活动名称、所属党组织、活动类型（会议/志愿活动/学习培训/其他）、活动内容描述、报名截止时间、开始结束时间、活动地点和最大参与人数等信息后保存至activity表，此时活动状态标记为待发布。管理员执行发布操作后状态变更为已发布且前端列表可见，用户可在报名截止时间前调用报名接口提交参与申请，系统在activity_enroll表中新增一条报名记录并将activity表的currentNum计数加一。活动开始后状态自动进入进行中阶段，参与者可通过扫码签到或手动签到两种方式完成考勤登记，signIn方法将enroll记录的isSign字段置为1。管理员在活动结束后执行结束操作触发updateActivityStatisticsById方法自动计算总报名人数、实际参与人数和签到率三项统计指标并回写至activity表的对应字段。

### 3.2.4 公告通知模块需求

系统应能够提供基于WangEditor富文本编辑器的公告内容编写能力，编辑器输出的HTML格式内容存储于notice表的content字段。公告管理应覆盖完整的状态流转——草稿态下仅创建者可见，执行发布操作后状态变为已发布并记录publishTime发布时间，已发布公告支持撤回恢复至草稿态、下架隐藏于列表前端、重新发布恢复可见性以及置顶排序提升展示优先级等多种管理操作。系统应支持公告附件的上传与下载功能，附件文件存储于static/notice/attachment/目录并在announcement_attachment表中建立关联记录，支持的文件格式包括doc/docx/pdf/xls/xlsx/ppt/pptx/txt/zip/rar共十种办公文档类型。阅读追踪功能应在用户打开公告详情时向announcement_read_record表插入一条含userId和readTime字段的阅读记录，未读数量统计接口实时返回当前用户的未读公告总数。分享链接功能通过生成带随机shareCode和expireTime过期时间的分享记录实现外部访问控制。

### 3.2.5 材料与文件管理模块需求

系统应能够支持材料模板文件的上传与管理功能，模板上传限制Word和PDF两种格式，文件经FileController接收后存储至static/template/目录并在file表中设置isTemplate标记为1。材料提交功能应允许用户选择对应的模板和发展阶段后上传个人材料文件，提交时在material_submission表中创建一条记录并设置初始auditStatus为待审核状态。多级审核流程应包含两个审批节点——初审由组织管理员执行，审核结果分为通过（进入终审队列）和退回（填写退回意见后状态变为已退回供用户修改后重新提交），终审由超级管理员执行，终审通过后材料进入归档状态。归档查询功能应按用户维度或组织维度展示历史已审核通过的材料清单及对应文件下载入口。统一文件上传服务应通过FileUploadBizEnum枚举区分USER_AVATAR头像、MATERIAL材料和TEMPLATE模板三种业务类型并为每种类型分配独立的存储子目录。

### 3.2.6 量化统计模块需求

系统应能够从用户和个人两个维度对党建活动数据进行量化指标计算与可视化呈现。用户维度的统计指标包括活动参与度（报名活动数除以总活动数的百分比）、签到率（已签到数除以报名数的百分比）和材料提交文件数量三项核心数据，计算结果持久化存储于user_quantify表的participateRate、signRate和fileCount字段。组织维度的统计指标涵盖活动次数、总报名人数、总签到人数、平均每次活动报名人数和平均每次活动签到人数五项汇总数据，存储于org_quantify表。定时任务QuantifyScheduledTask配置cron表达式0 0 1 * * ?实现每日凌晨1点自动触发执行，任务内部依次遍历全部Activity实体更新单活动的totalParticipant和actualParticipant统计值、聚合每位User的报名签到记录生成user_quantify日报数据、按orgId分组汇总各组织的平均值指标生成org_quantify日报数据。前端页面通过ECharts折线图展示指标的时间趋势变化，柱状图和饼图用于对比不同类别或不同组织的分布情况。

### 3.2.7 发展阶段管理模块需求

系统应能够对党员从入党积极分子到正式党员的全流程发展阶段进行数字化跟踪记录。每个阶段记录包含被培养人ID、培养人ID、阶段名称（积极分子/发展对象/预备党员/正式党员四种枚举值）、阶段起止时间、考察内容描述、阶段状态（0进行中/1已完成/2已终止）、考察结果（1合格/0不合格/2未审核）、审核人ID、审核时间和审核意见等完整属性字段。系统应支持阶段的增删改查基础操作以及提交审核和审核评定两类流程操作。关键操作的审计日志应通过AOP机制自动记录——@DevelopmentStageLogAnnotation注解标注于add、update、delete、submitForAudit和audit五个方法之上，DevelopmentStageLogAspect切面拦截注解方法调用后将操作类型（取自DevelopmentStageOperationEnum枚举的CREATE/UPDATE/DELETE/SUBMIT_AUDIT/AUDIT值）、操作人身份和操作时间戳写入development_stage_log日志表，确保阶段变更过程的可追溯性。

### 3.2.8 培养人关系管理模块需求

系统应能够建立和维护入党培养人与被培养人之间的对应关联关系，支持一名培养人同时培养多名被培养人的典型一对多业务场景。关系的建立通过TrainerRelationController的addTrainerRelation接口实现，接口接收被培养人ID和培养人ID后在trainer_relation表中新增一条记录并设置初始status为1表示关系进行中，同时写入startDate起始日期。创建时系统应校验同一userId和trainerId组合的唯一性约束uk_userId_trainerId防止重复建立关系。查询功能应提供三个维度——按被培养人ID查询其所有培养人列表、按培养人ID查询其负责的全部被培养人列表、按组织ID查询该组织内所有的培养关系集合。关系的解除操作通过delete接口将目标记录的status更新为3表示已终止并记录endDate结束日期。可选培养人列表接口应筛选出政治面貌符合要求的党员候选人供管理员选取。

### 3.2.9 组织关系转移模块需求

系统应能够支撑党员组织关系转移的线上申请与审批全流程。党员用户发起转移申请时需填写原所属组织、目标转入组织和转移原因三项必要信息，OrgRelationTransferController的addOrgRelationTransfer接口在org_relation_transfer表中创建一条记录并自动设置approveStatus为PENDING待审批状态（对应ApproveStatusEnum枚举值1）。管理员收到待办提醒后查看申请详情并可执行审批操作，传入APPROVED状态值（枚举值2）表示同意转移则系统将该党员user表的orgId字段更新为目标组织ID；传入REJECTED状态值（枚举值3）表示拒绝转移则记录拒绝原因。无论审批结果如何均需在记录中补充approveUserId审批人ID、approveTime审批时间戳和approveComment审批意见文本。历史记录查询应分别提供按党员ID查询其全部转移履历和按组织ID查询涉及该组织的所有转移记录两种视角的数据检索接口。

## 3.3 本章小结

本章从技术、经济和操作三个角度论证了智慧党建管理系统开发的可行性，确认所选技术栈成熟可靠、开发成本低廉、操作门槛适度。随后围绕九大功能模块逐一阐述了系统的功能性需求规格——用户管理与五类角色权限控制、党组织三级树形层级管理、活动全生命周期数字化流转、公告通知的富文本发布与多状态管理、材料模板与多级审核流程、双维度量化统计自动计算、入党发展全流程AOP审计跟踪、一对多培养人关系维护以及组织关系转移在线审批。上述需求覆盖了高校党建工作的主要业务场景，明确了系统各模块的功能边界和交互逻辑，为下一章的系统总体设计和数据库建模提供了清晰的输入依据。
